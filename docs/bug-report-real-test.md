## ۱. خطا در ساخت ادمین جدید (بحرانی)

**بخش:** پنل ادمین › مدیریت ادمین‌ها  
**وضعیت:** تأیید شده از طریق کد

### شرح مشکل

هر سه نقش کارشناس، ناظر رسانه‌ای و تحلیلگر بازرگانی هنگام ساخت خطا می‌دهند.

### علت احتمالی — اول: خطای validation بدون راهنما

کد server-side برای `createAdmin` درست است. اما `onError` در فرانت‌اند فقط `err.message` را در toast نشان می‌دهد، نه جزئیات خطاهای فیلدها:

```ts
// apps/web/src/routes/_authenticated/admin/admins.tsx
onError: (err) => toast.error(err.message),
```

اگر ورودی اشتباه باشد (مثلاً موبایل بدون `09` یا یوزرکد کمتر از ۴ کاراکتر)، zod validation خطا می‌دهد اما کاربر نمی‌فهمد کدام فیلد مشکل دارد.

### علت احتمالی — دوم: انقضای توکن

توکن دسترسی ۱۵ دقیقه اعتبار دارد. اگر SUPER_ADMIN بیش از ۱۵ دقیقه روی صفحه مانده باشد بدون رفرش، فراخوانی API با UNAUTHORIZED رد می‌شود. در این حالت toast پیام `err.message` را نشان می‌دهد که از نظر کاربر شبیه «خطا» است اما در واقع session منقضی شده.

### راه حل

**الف — بهبود نمایش خطا در فرم (فوری):**
```ts
onError: (err) => {
  const zodErrors = err.data?.zodError?.fieldErrors;
  if (zodErrors) {
    const firstError = Object.values(zodErrors).flat()[0];
    toast.error(firstError ?? err.message);
  } else {
    toast.error(getFriendlyTrpcError(err));
  }
},
```

**ب — اضافه کردن validation سمت کلاینت:**
با `react-hook-form` و `zodResolver` در فرم `admins.tsx` تا قبل از ارسال، خطاها روی فیلد مربوطه نمایش داده شوند.

**ج — رفع مشکل انقضای توکن (مطابق باگ ۴)**

## ۲. از دست رفتن داده پس از انقضای session در فرم طولانی (بحرانی)

**بخش:** ثبت محصول جدید  
**منبع:** گزارش صفحه ۹ از `prob2.pdf`  
**وضعیت:** تأیید شده از طریق کد

### شرح مشکل

کاربر پس از چند دقیقه پر کردن فرم، با کلیک «ثبت» به صفحه ورود هدایت می‌شود و تمام داده‌های وارد شده از دست می‌رود.

### ریشه فنی

```
JWT_ACCESS_EXPIRY=15m   ← توکن ۱۵ دقیقه اعتبار دارد
JWT_REFRESH_EXPIRY=7d   ← refresh token 7 روز (در cookie httpOnly)
```

**رفرش توکن فقط هنگام mount صفحه انجام می‌شود:**
```ts
// apps/web/src/routes/_authenticated.tsx
useEffect(() => {
  if (!isAuthenticated) { void navigate({ to: '/auth/login' }); return; }
  if (!accessToken && !refreshMutation.isPending) { refreshMutation.mutate(); }
  // ← هیچ interval یا timer برای رفرش دوره‌ای وجود ندارد
}, [isAuthenticated]);
```

**tRPC client هیچ interceptorی برای retry پس از 401 ندارد:**
```ts
// apps/web/src/trpc.ts
httpBatchLink({
  url: '/trpc',
  headers() { ... }
  // ← اگر 401 برگردد، retry نمی‌کند
})
```

در نتیجه: پس از ۱۵ دقیقه، هر mutation (ثبت محصول، ثبت RFQ، ...) با خطای UNAUTHORIZED شکست می‌خورد و چون route guard فعال نیست در آن لحظه، کاربر روی صفحه گیر می‌کند.

### راه حل

**الف — راه‌حل کوتاه‌مدت: رفرش خودکار دوره‌ای**
```ts
// در AuthenticatedLayout — رفرش ۱ دقیقه قبل از انقضا
useEffect(() => {
  const REFRESH_INTERVAL = 13 * 60 * 1000; // هر ۱۳ دقیقه
  const interval = setInterval(() => {
    refreshMutation.mutate();
  }, REFRESH_INTERVAL);
  return () => clearInterval(interval);
}, []);
```

**ب — راه‌حل صحیح‌تر: custom tRPC link با retry پس از 401**
```ts
// tRPC link که روی UNAUTHORIZED توکن را رفرش و retry می‌کند
import { retryLink } from '@trpc/client';
```
پیاده‌سازی: custom link که خطای UNAUTHORIZED را intercept کند، `auth.refreshToken` را صدا بزند، توکن جدید را ذخیره کند و request را retry کند.

**ج — نمایش هشدار انقضای session**
قبل از انقضا، یک toast warning نشان دهد: «جلسه شما در ۲ دقیقه منقضی می‌شود.»

## ۳. validation فیلدهای درصد (پیش‌پرداخت + تحویل) نمایش داده نمی‌شود (متوسط)

**بخش:** ثبت محصول جدید › مرحله ۲ › شرایط فروش  
**منبع:** گزارش صفحه ۲ از `prob2.pdf`  
**وضعیت:** تأیید شده از طریق کد

### شرح مشکل

دو فیلد «درصد پیش‌پرداخت» و «درصد هنگام تحویل» باید جمعشان ۱۰۰ شود. validation در schema وجود دارد اما پیام خطا نشان داده نمی‌شود.

### ریشه فنی

در `product.schema.ts`:
```ts
export const saleConditionsSchema = z.object({
  advancePercent: z.number().int().min(0).max(100),
  onDeliveryPercent: z.number().int().min(0).max(100),
}).refine(
  (d) => d.advancePercent + d.onDeliveryPercent === 100,
  { message: 'جمع درصد پیش‌نقد و تحویل باید ۱۰۰ باشد' },
);
```

اما `saleConditions` در `createProductSchema` به صورت `optional()` تعریف شده است:
```ts
saleConditions: saleConditionsSchema.optional(),
```

وقتی `saleConditions` به صورت optional است، اگر هر دو فیلد وارد شوند اما جمعشان ۱۰۰ نشود، خطای `refine` در مسیر `['saleConditions']` تولید می‌شود. اما در فرم:
```tsx
{(errors as any).saleConditions?.message && (
  <p className="field-error col-span-2">{(errors as any).saleConditions.message}</p>
)}
```

`react-hook-form` ممکن است خطای `refine` روی object nested optional را در `errors.saleConditions.message` قرار ندهد — بلکه در `errors.saleConditions.root?.message` یا جای دیگر.

### راه حل

**الف — استفاده از `errors.saleConditions?.root?.message`:**
```tsx
{(errors.saleConditions?.root?.message || (errors as any).saleConditions?.message) && (
  <p className="field-error col-span-2">
    {errors.saleConditions?.root?.message ?? (errors as any).saleConditions?.message}
  </p>
)}
```

**ب — validation مستقل در فرم (مطمئن‌ترین راه):**
از `watch` برای خواندن مقادیر لحظه‌ای و نمایش خطا به صورت controlled استفاده شود:
```tsx
const advance = watch('saleConditions.advancePercent');
const onDelivery = watch('saleConditions.onDeliveryPercent');
const percentError = (advance + onDelivery) !== 100 && advance !== undefined && onDelivery !== undefined
  ? 'جمع درصد پیش‌پرداخت و تحویل باید ۱۰۰ باشد'
  : null;
```

## ۶. نمایش خطای Zod validation در mutation های admin (متوسط)

**بخش:** تمام mutation های admin  
**وضعیت:** تأیید شده از طریق کد

در تمام mutation های admin (createAdmin, updateAdminRole, removeAdmin) فقط `err.message` نمایش داده می‌شود:

و در صورت نیاز، بهبود `getFriendlyTrpcError` برای خواندن اولین field error از `err.data.zodError`.

## ۷. نمایش «undefined» به جای واحد مقدار در کارت RFQ (بحرانی)

## ۱۳. سیستم ثبت کالای دوم را بلوک می‌کند بدون پیام راهنما (متوسط)

**بخش:** ثبت کالای جدید  
**منبع:** صفحه ۱۲ از `prob2.pdf`  
**وضعیت:** تأیید شده از روی تصویر

### شرح مشکل

پس از ثبت اولین کالا، دکمه «+ کالای جدید» فعال است اما کلیک روی آن به نتیجه نمی‌رسد یا خطایی می‌دهد — کاربر نمی‌فهمد چرا نمی‌تواند کالای دوم را ثبت کند. علت واقعی این است که تا زمانی که کالای قبلی توسط ادمین تایید نشده، ثبت کالای جدید مجاز نیست. اما این محدودیت به کاربر توضیح داده نمی‌شود.

**نکته تستر:** «ممکن است یک شرکت همزمان تصمیم به ثبت چندین کالا داشته باشد» — این یک feature request هم هست که اجازه ثبت موازی چند کالا داده شود.

### راه حل

**کوتاه‌مدت:** نمایش پیام راهنما هنگام کلیک روی «کالای جدید» در صورت وجود کالای در حال بررسی:  
> «کالای شما در حال بررسی توسط ادمین است. پس از تایید می‌توانید کالای جدید ثبت کنید.»

**بلندمدت:** بررسی امکان‌پذیری اجازه به ثبت موازی چند کالا.

**وضعیت:** تأیید شده از روی تصویر

### شرح مشکل

پس از ثبت کالا یا پس از تایید توسط ادمین، صفحه «کالاهای من» وضعیت جدید را فوراً منعکس نمی‌کند. کاربر مجبور است چندین بار صفحه را رفرش کند یا دوباره لاگین کند تا تغییرات نمایش داده شوند.

### علت احتمالی

- `staleTime` کوئری TanStack Query برای لیست کالاها خیلی طولانی تنظیم شده
- Mutation پس از موفقیت، `invalidateQueries` لازم را صدا نمی‌زند

### راه حل

پس از موفقیت mutation ثبت کالا و mutation تایید کالا، کوئری لیست کالاها invalidate شود:
```ts
onSuccess: () => {
  void utils.product.list.invalidate();
}
```

### شرح مشکل

کاربر یک درخواست تبلیغاتی (بنر) ثبت کرد. سیستم پیام داد «درخواست تبلیغ ثبت شد و در انتظار بررسی است». پس از تایید ادمین، آگهی تایید شده هیچ جایی در سایت برای کاربران عادی نمایش داده نمی‌شود — نه در داشبورد، نه در Marketplace، نه در صفحه جداگانه‌ای. پنل ادمین ۳ آگهی تایید شده دارد اما هیچ صفحه عمومی برای نمایش آن‌ها وجود ندارد.

### راه حل

طراحی و پیاده‌سازی صفحه عمومی نمایش آگهی‌ها (مشابه صفحه Marketplace اما برای تبلیغات).
