# نقشه راه پیاده سازی بر اساس درخواست جدید کارفرما

تاریخ: 2026-05-03
وضعیت: آماده اجرا
مخاطب: Product, UI/UX, Frontend, Backend, QA

## 1) هدف این نقشه راه

هم راستا کردن تجربه محصول با وایرفریم/سایت مپ پیشنهادی کارفرما، با حداقل ریسک برای معماری فعلی پروژه.

اصل کلیدی:
- تغییر در Presentation و Navigation اولویت دارد.
- هسته فنی (معماری، مدل داده، API) فقط در صورت نیاز واقعی تغییر می کند.

## 2) ترجمه درخواست کارفرما به ساختار فعلی پروژه

نگاشت منو و صفحات:
- Dashboard -> مسیر فعلی dashboard
- کالاها -> مسیر فعلی products
- درخواست ها (RFQ) -> مسیر فعلی trade
- Marketplace (بازار) -> مسیر فعلی catalog
- پیام ها -> مسیر فعلی chat
- AI مشاور -> فاز MVP روی chat با حالت مشاور
- اسناد -> مسیر فعلی downloads
- مالی -> صفحه جدید placeholder در فاز اول

نکته اجرایی:
- به جای شکستن مسیرهای فعلی، ابتدا alias مسیرها و تغییر label انجام شود.

## 3) محدوده MVP این تغییر

در محدوده:
- هم راستاسازی Navigation (Top/Side)
- بازطراحی Dashboard بر اساس کارت ها + پیشنهاد AI + فرصت ها
- همسان سازی واژگان UI با ادبیات کارفرما (RFQ, Marketplace, AI مشاور)
- آماده سازی صفحات Placeholder برای بخش هایی که هنوز دامنه قطعی ندارند (مالی)
- به روزرسانی تست های E2E مسیرهای ناوبری و سناریوهای اصلی

خارج از محدوده این موج:
- بازنویسی کامل دامنه پیام رسان یا AI
- تغییر بنیادین Auth/Role model
- افزودن قابلیت های مالی واقعی (تراکنش، کیف پول، تسویه)

## 4) فازبندی اجرایی

### فاز 0: هم راستاسازی محصول و واژگان (1 روز)

کارها:
- تایید رسمی نگاشت منوها با کارفرما
- نهایی سازی نام گذاری ها در فارسی/انگلیسی
- نهایی سازی مسیرهای alias

خروجی:
- جدول مصوب As-Is / To-Be
- لیست مسیرهای alias

DoD:
- Product Owner و تیم فنی روی نگاشت نهایی توافق کتبی داشته باشند.

### فاز 1: ناوبری و Site Map قابل استفاده (2 تا 3 روز)

کارها:
- به روزرسانی Sidebar و Mobile Nav
- افزودن alias route برای:
  - rfq -> trade
  - marketplace -> catalog
  - messages -> chat
  - ai-advisor -> chat
  - documents -> downloads
- افزودن صفحه placeholder برای finance

خروجی:
- منوی جدید مطابق متن کارفرما
- بدون شکستن مسیرهای قدیمی

DoD:
- کاربر با هر یک از اصطلاحات جدید به صفحه مورد انتظار برسد.
- هیچ رگرسیون مسیر برای لینک های قدیمی رخ ندهد.

### فاز 2: بازطراحی Dashboard MVP (3 تا 4 روز)

کارها:
- بخش خوش آمدگویی و کارت های KPI:
  - تعداد کالا
  - درخواست فعال
  - پیام جدید
- پنل پیشنهادهای AI (در MVP از داده ساده یا خلاصه)
- پنل فرصت های جدید (feed قابل مشاهده)
- بهبود Responsive موبایل/تبلت

خروجی:
- داشبورد نزدیک به وایرفریم کارفرما با داده واقعی یا نیمه واقعی

DoD:
- KPI ها قابل مشاهده و پایدار باشند.
- Dashboard روی دسکتاپ و موبایل بدون شکست UI رندر شود.

### فاز 3: هم راستاسازی صفحات حوزه تجاری (4 تا 6 روز)

کارها:
- Products:
  - واضح تر کردن فیلتر/جستجو/لیست
  - همسان سازی عنوان ها با ادبیات کارفرما
- RFQ (Trade):
  - label و CTA ها با زبان RFQ
  - ورودی های مهم در دسترس سریع کاربر
- Marketplace:
  - تاکید بر کارت کالا + فیلترهای کلیدی
- Messages + AI Advisor:
  - یک ورودی برای پیام ها
  - یک ورودی برای مشاور با state متفاوت روی همان بستر

خروجی:
- صفحات کلیدی قابل نمایش به کارفرما با زبان مشترک

DoD:
- سناریوهای اصلی خرید/فروش/پیام در UI قابل انجام باشند.
- واژگان کلیدی در کل مسیرها یکدست باشند.

### فاز 4: تست، تثبیت و تحویل دمو (2 تا 3 روز)

کارها:
- به روزرسانی E2E برای مسیرها و label های جدید
- Smoke تست تمام alias ها
- تست ریسپانسیو برای موبایل اندروید
- آماده سازی چک لیست دمو

خروجی:
- دمو قابل ارائه به کارفرما با ریسک پایین

DoD:
- تست های بحرانی پاس باشند.
- مسیرهای قدیم و جدید همزمان سالم باشند.

## 5) بک لاگ اجرایی (اولویت بندی)

P0:
- به روزرسانی Navigation
- alias route ها
- بازطراحی Dashboard MVP
- تست E2E مسیرهای ناوبری

P1:
- همسان سازی labels در Products/RFQ/Marketplace
- Placeholder مالی
- حالت AI Advisor روی صفحه chat

P2:
- بهبودهای ظاهری پیشرفته (animation/polish)
- ارتقای AI Advisor از mock به داده زنده تکمیلی

## 6) ریسک ها و برنامه کاهش ریسک

ریسک: شکستن لینک های قبلی
- کاهش: حفظ مسیر قدیمی + افزودن alias

ریسک: اختلاف تفسیر واژگان بین تیم و کارفرما
- کاهش: تایید واژگان در فاز 0 و freeze اصطلاحات

ریسک: افزایش حجم کار فراتر از MVP
- کاهش: محدوده صریح In-Scope/Out-of-Scope و پذیرش مرحله ای

## 7) برنامه شروع پیاده سازی (Sprint 1 پیشنهادی)

روز 1:
- فاز 0 کامل
- شروع Navigation refactor

روز 2:
- تکمیل alias routes + placeholder مالی
- بازبینی داخلی UX

روز 3:
- Dashboard MVP (KPI + AI panel + فرصت ها)

روز 4:
- رفع رگرسیون + تست E2E مسیرها

روز 5:
- دمو داخلی + اصلاحات نهایی + آماده سازی ارائه کارفرما

## 8) معیار پذیرش نهایی این موج

- کارفرما منو و صفحات را با واژگان مورد نظر خود ببیند.
- مسیرهای اصلی محصول فعلی بدون شکست باقی بمانند.
- دمو MVP با تست های بحرانی پاس قابل ارائه باشد.
- مسیر ادامه فاز بعدی (مالی واقعی و AI پیشرفته) شفاف و جداگانه تعریف شده باشد.

## 9) ادامه اولویت بندی (پس از تحویل MVP فعلی)

### 9.1 اولویت های فوری (P0 - این اسپرینت)

- پایداری محیط تست و دمو:
  - یکسان سازی پورت پیش فرض E2E و محیط dev
  - ثبت Runbook کوتاه برای بالا آوردن infra + app + seed دیتابیس
- تثبیت ورود/احراز هویت در سناریوهای بحرانی:
  - بررسی و پوشش E2E برای مسیر ورود با نقش های اصلی
  - پایدارسازی داده های seed برای جلوگیری از flaky شدن تست ها
- چک لیست دمو کارفرما:
  - مسیر نمایشی Dashboard -> Products -> RFQ -> Marketplace -> Messages/AI Advisor
  - معیار pass/fail دمو قبل از ارائه

خروجی P0:
- اجرای قابل تکرار دمو روی محیط توسعه بدون خطای زیرساختی
- گزارش سبز تست های بحرانی مرتبط با ناوبری و ورود

### 9.2 اولویت های نزدیک (P1 - اسپرینت بعد)

- Finance Placeholder نسخه قابل ارائه:
  - تعریف محتوای اولیه صفحه (وضعیت مالی، درگاه های آینده، پیام راهنما)
  - مسیرهای ناوبری مرتبط از Dashboard/Sidebar
- یکپارچگی واژگان در کل UI:
  - بازبینی متن دکمه ها، headingها و breadcrumbها در صفحات باقیمانده
  - همسان سازی متن خطاها و وضعیت ها با ادبیات کارفرما
- Dashboard Data Quality:
  - اتصال KPIها به داده قابل اتکا (با fallback مناسب)
  - نمایش empty/loading/error state استاندارد

خروجی P1:
- محصول از حالت MVP صرف به نسخه نمایشی پایدارتر برای تصمیم گیری کارفرما برسد.

### 9.3 اولویت های بعدی (P2 - پس از تایید کارفرما)

- AI Advisor مرحله 2:
  - ارتقا از پاسخ های ساده به پیشنهادهای مبتنی بر داده واقعی کاربر
  - تعریف KPI برای سنجش کیفیت پیشنهادها
- بهبود UX پیشرفته:
  - polish بصری، motion هدفمند، و بهینه سازی تجربه موبایل
- آماده سازی دامنه مالی واقعی:
  - تحلیل دامنه، مدل داده، و APIهای مورد نیاز بدون ورود به پیاده سازی کامل

خروجی P2:
- backlog فاز توسعه عمیق با برآورد زمان و ریسک مشخص برای قرارداد بعدی

### 9.4 ترتیب پیشنهادی اجرا (Dependency Order)

1. Runbook محیط + پایداری تست
2. چک لیست دمو و تثبیت احراز هویت
3. Finance Placeholder و یکپارچگی کامل واژگان
4. کیفیت داده داشبورد
5. AI Advisor مرحله 2 و UX polish

### 9.5 معیار خروج برای شروع فاز بعد

- حداقل یک اجرای کامل E2E بحرانی بدون شکست محیطی
- تایید کارفرما روی واژگان و ناوبری جدید
- آماده بودن Finance Placeholder برای دمو
- وجود backlog اولویت بندی شده با Owner و زمان بندی تقریبی

## 10) چک لیست دمو (Pass/Fail)

### 10.1 آماده سازی پیش از دمو

- [ ] Docker infra بالا و healthy باشد (postgres, redis, minio)
- [ ] backend روی 4000 و frontend روی 3002 در دسترس باشند
- [ ] کاربر seed قابل ورود باشد (userCode: 0000001)
- [ ] آخرین اجرای تست های بحرانی بدون خطا باشد

### 10.2 سناریوی نمایشی اصلی

1. ورود کاربر:
  - Pass: ورود با userCode و رمز انجام شود و کاربر به Dashboard/Profile برسد.
  - Fail: timeout یا ماندن در صفحه ورود.
2. ناوبری اصلی:
  - Pass: آیتم های Dashboard, Products, RFQ, Marketplace, Messages, AI Advisor, Documents, Finance در Sidebar دیده شوند.
  - Fail: برچسب یا مسیر اشتباه.
3. alias routes:
  - Pass: rfq -> trade ، marketplace -> catalog ، documents -> downloads ، messages -> chat ، ai-advisor -> chat.
  - Fail: ریدایرکت ناقص یا صفحه نادرست.
4. Dashboard MVP:
  - Pass: KPI ها، پیشنهادهای AI و فرصت های جدید نمایش داده شوند (حتی در حالت empty/error استاندارد).
  - Fail: شکست رندر یا نبود state مناسب.
5. RFQ جریان اصلی:
  - Pass: ایجاد RFQ جدید با فیلدهای کلیدی و نمایش در لیست.
  - Fail: بسته نشدن فرم یا نبود آیتم جدید.
6. Chat و AI Advisor:
  - Pass: در messages حالت پیام ها و در ai-advisor حالت مشاور به درستی فعال شود.
  - Fail: مود اشتباه یا نبود CTAهای اصلی.
7. Finance Placeholder:
  - Pass: صفحه مالی با پیام وضعیت، قابلیت های آینده و مسیرهای جایگزین فعال نمایش داده شود.
  - Fail: صفحه خالی یا ناهمخوان با سناریوی دمو.

### 10.3 خروجی مورد انتظار پس از دمو

- [ ] لیست بازخورد کارفرما به تفکیک: واژگان، ناوبری، UI، نیازمندی جدید
- [ ] تصمیم کارفرما برای شروع P1 ثبت شود
- [ ] backlog نسخه بعد با owner و ETA به روز شود
