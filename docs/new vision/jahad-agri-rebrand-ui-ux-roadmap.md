## هدف سند
این سند برای مرحله قبل از پیاده سازی تهیه شده است. هدف آن این است که تیم فنی و کارفرما قبل از دست زدن به کد، روی دامنه پاکسازی برند، زبان محصول، تم بصری، الگوی تجربه کاربری و ترتیب اجرای تغییرات توافق کنند.

این سند جایگزین نقشه راه فنی اصلی نیست؛ بلکه مکمل سند `jahad-agri-implementation-roadmap.md` است و به صورت خاص روی دو مسئله تمرکز دارد:

- حذف ردپاهای قابل مشاهده و اثرگذار برند قبلی «تاجر هوشمند» از نسخه جهاد کشاورزی.
- تعریف مسیر بازطراحی UI/UX با الهام کنترل شده از Tridge، بدون کپی مستقیم محتوا، کد، دارایی یا چیدمان اختصاصی آن سایت.

## جمع بندی وضعیت فعلی
پروژه از نظر مدل های کشاورزی، routeهای harvest و market insight، taxonomy شغلی و بخشی از متن های دامنه کشاورزی جلو رفته است. با این حال هنوز در چند سطح ردپای محصول قبلی دیده می شود:

- برند قابل مشاهده در صفحه ورود، ثبت نام، سایدبار، هدر موبایل، صفحات درباره/تماس/خبر و metadata مرورگر.
- واژگان عمومی تجارت، صادرات/واردات، ترمینال تجاری، RFQ و Marketplace در بخش هایی از UI.
- system prompt هوش مصنوعی که هنوز خود را دستیار «مرکز تجارت متمرکز هوشمند ایرانیان» معرفی می کند.
- subject و متن ایمیل هایی که هنوز «پلتفرم تجارت هوشمند» را استفاده می کنند.
- تم بصری آبی/امبر/تیره و heroهای نارنجی/تجاری که با هویت کشاورزی و خواسته جدید فاصله دارند.

## محدوده پاکسازی مرحله اول
در این مرحله، تغییرات باید روی سطح هایی انجام شود که برای کاربر، کارفرما یا ارتباطات بیرونی قابل مشاهده اند. تغییر نام های داخلی و قراردادهای API باید فعلا با احتیاط انجام شود.

### داخل محدوده
| سطح | نمونه تغییر |
|---|---|
| UI عمومی | صفحه اصلی، درباره، تماس، خبر، کاتالوگ عمومی، harvest و market insight |
| UI احراز هویت | ورود، ثبت نام، OTP، پیام های کپچا و خطا |
| Layout کاربر | سایدبار، هدر موبایل، bottom nav، نام کوتاه برند و آیکون برند |
| داشبورد | عنوان کارت ها، پیشنهادهای AI، فرصت ها، آمارها و CTAها |
| پنل ادمین | عنوان ماژول ها، توضیح گزارش ها، صف مدارک، taxonomy و محتوا |
| metadata | title، description، favicon/manifest در صورت وجود |
| AI | system prompt، quick prompts، متن placeholderها و guardrailها |
| ایمیل | subject، CTA، رنگ دکمه و متن های welcome/document/trade matched |
| i18n | کلیدهای قابل مشاهده فارسی/انگلیسی/عربی، با اولویت فارسی |

### خارج از محدوده مرحله اول
| سطح | دلیل تعویق |
|---|---|

| نام routerها مانند `tradeRouter` | تغییر route/API داخلی blast radius بالاتری دارد و باید جداگانه طراحی شود. |
| مدل های DB مانند `TradeRequest` | rename دیتابیسی نیازمند migration و strategy سازگاری است. |
| route pathهای تثبیت شده مانند `/rfq` | تغییر URL باید با redirect، تست E2E و تصمیم SEO/UX انجام شود. |
| داده های historical قدیمی | نیازمند migration محتوایی و تصمیم مالک داده است. |

## راهبرد واژگان محصول
هدف این است که UI از زبان «بازرگانی عمومی» به زبان «اکوسیستم کشاورزی، اعتبارسنجی، محصول، خدمت، فرصت و داده بازار» منتقل شود.

### جدول جایگزینی پیشنهادی
| عبارت/مفهوم قدیمی | عبارت پیشنهادی در UI | توضیح |
|---|---|---|
| تجارت هوشمند | سامانه یکپارچه ذینفعان حوزه کشاورزی | نام اصلی در header، auth و metadata |
| سامانه تجارت ایرانیان | شبکه تخصصی ذینفعان و فرصت های کشاورزی | subtitle کوتاه تر برای فضاهای محدود |
| مرکز تجارت متمرکز هوشمند ایرانیان | سامانه یکپارچه ذینفعان حوزه کشاورزی | برای prompt، ایمیل و metadata |
| تاجر | فعال کشاورزی / ذینفع کشاورزی | فقط در UI؛ enum داخلی فعلا تغییر نکند. |
| Marketplace | بازار محصولات | در صورت نیاز: بازار محصولات و خدمات کشاورزی |
| RFQ | درخواست/فرصت | در متن های آموزشی می توان «درخواست قیمت» را هم استفاده کرد. |
| Trade Request | درخواست کشاورزی / فرصت همکاری | بسته به buy/sell/service/collaboration |
| کالا | محصول / خدمت | در حوزه کشاورزی «محصول» پیش فرض باشد؛ برای خدمات جدا شود. |
| ترمینال صادرات و واردات | لایه داده و فرصت های کشاورزی | عبارت ترمینال حس محصول قبلی دارد. |
| تحلیل بازرگانی | تحلیل بازار کشاورزی | با disclaimer نمایشی/غیرتعهدی |
| اخبار تجاری | اخبار و آگاهی های کشاورزی | برای بخش news |
| خریدار/فروشنده عمومی | متقاضی، عرضه کننده، تولیدکننده، بهره بردار | بر اساس context صفحه |

### قواعد نگارشی
- در صفحه های فارسی، از واژه های انگلیسی فقط وقتی استفاده شود که برای مخاطب حرفه ای ضروری است.
- `AI` در متن عمومی بهتر است «دستیار هوشمند» یا «مشاور هوشمند» باشد.
- «جهاد کشاورزی» فقط در جایی استفاده شود که از نظر حقوقی/برندی مجاز و تایید شده است؛ در غیر این صورت «حوزه کشاورزی» امن تر است.
- واژگان صادرات و واردات حذف کامل نشوند، ولی در جایگاه زیرمجموعه تجارت کشاورزی قرار بگیرند نه هویت اصلی صفحه.

## تحلیل کنترل شده از Tridge
نسخه دانلود شده Tridge برای الهام بصری بررسی شد. در مرور محتوایی، الگوی اصلی صفحه شامل این بخش هاست:

- هدر تیره، مینیمال و CTAهای روشن.
- hero با تیتر کوتاه، ادعای داده محور و چند شاخص عددی.
- نمایش اعتماد/شبکه با لوگوها یا شاخص های اجتماعی.
- سکشن «چالش» برای بیان مشکل صنعت.
- سکشن «چشم انداز/راهکار» با زبان داده، شبکه و AI.
- کارت های راهکار برای گروه های کاربری متفاوت.
- مقایسه و اثبات ارزش با جدول/کارت های داده محور.

### مواردی که باید الهام گرفته شود
| الگو | تطبیق برای سامانه کشاورزی |
|---|---|
| Hero داده محور | نمایش تعداد محصولات، دسته های شغلی، استان ها، مجوزهای در انتظار و فرصت های فعال |
| چالش صنعت | پراکندگی اطلاعات، سختی اعتبارسنجی، دشواری یافتن طرف معتبر، نبود داده برداشت/بازار |
| راهکار لایه ای | هویت و اعتماد، بازار و فرصت، داده و آگاهی، دستیار هوشمند |
| کارت های گروه کاربری | برای تولیدکننده، بازرگان، کارشناس/خدمت رسان، مدیر/اپراتور |
| هدر تیره | منوی کوتاه، CTA ثبت نام/ورود، زبان، بدون شلوغی بصری |
| زبان اجرایی | جمله های کوتاه، ارزش عملی، تمرکز بر workflow |

### مواردی که نباید کپی شود
- متن های انگلیسی Tridge، ادعاهای عددی، نام محصولات و ساختار تجاری اختصاصی آن.
- تصاویر، لوگوها، CSS تولیدی، کامپوننت ها یا ترتیب دقیق سکشن ها به شکل یکسان.
- ادعاهایی مانند تعداد بازار، کاربر یا دیتاپوینت مگر با داده واقعی یا seed نمایشی تایید شده.
- تبدیل صفحه به کپی بازاریابی؛ محصول شما باید سامانه تخصصی ذینفعان کشاورزی باشد، نه clone یک سایت خارجی.

## معماری پیشنهادی تجربه کاربری

### مسیرهای عمومی
| مسیر | هدف UX | نکات طراحی |
|---|---|---|
| صفحه اصلی | معرفی سریع ارزش سامانه و هدایت به ثبت نام/بازار/داده ها | hero داده محور، چالش/راهکار، گروه های ذینفع، CTA محدود |
| بازار محصولات | جست وجوی محصولات و خدمات کشاورزی | فیلترهای شغلی، محصول، استان، وضعیت اعتبار، فصل برداشت |
| تقویم برداشت | نمایش آگاهی فصلی | نقشه/فیلتر منطقه، disclaimer واضح، رنگ بندی ماه ها |
| تحلیل بازار | محتوای نمایشی و شاخص ها | کارت های داده، برچسب نوع تحلیل، تاریخ انتشار، عدم تعهد |
| درباره | توضیح ماموریت و دامنه سامانه | حذف روایت انجمن صادرات/واردات قبلی، تمرکز بر اکوسیستم کشاورزی |
| تماس | ارتباط رسمی، پشتیبانی و دریافت پیام | فرم ساده، اطلاعات سازمانی تایید شده، نظرسنجی در صورت نیاز |

### مسیرهای کاربر احراز شده
| مسیر | هدف UX | نکات طراحی |
|---|---|---|
| داشبورد | نمای عملیاتی روزانه | تکمیل پروفایل، وضعیت مجوز، فرصت های جدید، اعلان ها، تقویم برداشت |
| پروفایل تخصصی | تکمیل هویت و دسته بندی ذینفع | progress روشن، دسته شغلی چندگانه، موقعیت، نوع شخصیت |

| درخواست/فرصت | ثبت نیاز یا فرصت همکاری | فرم چندنوعی: خرید، فروش، خدمت، مشاوره، همکاری |
| مجوزها | بارگذاری و پیگیری اعتبارسنجی | وضعیت، توضیح رد/اصلاح، تاریخچه |
| پیام ها | پیگیری تعاملات | اولویت بندی موارد نیازمند اقدام |
| دستیار هوشمند | راهنمای کمکی | پاسخ محدود، بدون تصمیم قطعی، context-aware |

| تحلیل بازار | محتوای تحلیلی | draft/published، منبع، disclaimer |
| گزارش ها | گزارش پایه مدیریتی | نمودارهای ساده و جدول های قابل export در فاز بعد |

## ساختار پیشنهادی صفحه اصلی آینده
صفحه اصلی باید اولین viewport را با نام محصول و ارزش دامنه کشاورزی شروع کند. پیشنهاد ساختار:

1. هدر تیره و مینیمال
   - لوگو/نام کوتاه
   - لینک های اصلی: بازار محصولات، تقویم برداشت، تحلیل بازار، درباره، تماس
   - CTA: ورود، ثبت نام

2. Hero داده محور
   - H1: «سامانه یکپارچه ذینفعان حوزه کشاورزی»
   - متن پشتیبان: «یک بستر تخصصی برای ثبت پروفایل، اعتبارسنجی اولیه، معرفی محصول و خدمت، ثبت درخواست و مشاهده داده های آگاه ساز کشاورزی.»
   - CTA اصلی: «شروع عضویت»
   - CTA ثانویه: «مشاهده بازار محصولات»
   - شاخص ها: دسته های شغلی، محصولات ثبت شده، استان ها، مجوزهای در انتظار بررسی، تحلیل های منتشرشده

3. نوار اعتماد/داده
   - به جای لوگوهای مشتری، از داده های داخلی یا seed تایید شده استفاده شود.
   - مثال: «taxonomy شغلی»، «مجوزهای قابل بررسی»، «داده های تقویم برداشت».

4. چالش های اکوسیستم کشاورزی
   - پراکندگی اطلاعات ذینفعان
   - دشواری اعتبارسنجی مجوزها
   - نبود مسیر یکپارچه برای درخواست و فرصت
   - کمبود آگاهی فصلی و بازار در یک مکان

5. راهکار سامانه
   - لایه هویت و اعتماد
   - لایه بازار و تعامل
   - لایه داده و آگاهی
   - لایه دستیار هوشمند کمکی

6. گروه های کاربری
   - تولیدکننده/کشاورز/باغدار
   - بازرگان/صادرکننده/واردکننده کشاورزی
   - شرکت خدماتی و کارشناس تخصصی
   - اپراتور و مدیر سامانه

7. بخش محصولات و فرصت ها
   - کارت های واقعی از API در صورت وجود
   - empty state حرفه ای در نبود داده

8. تحلیل بازار و تقویم برداشت
   - دو کارت/باند جداگانه با CTA
   - disclaimer کوتاه و روشن

9. CTA پایانی
   - «پروفایل تخصصی خود را تکمیل کنید» یا «به شبکه ذینفعان کشاورزی بپیوندید»

## فازبندی اجرایی بازطراحی

## فاز 0: Freeze تصمیم های برند و دامنه
هدف: جلوگیری از تغییرات پراکنده در طول پیاده سازی.

### کارها
- تایید نام نمایشی نهایی در همه زبان ها.
- تایید استفاده یا عدم استفاده از نام «جهاد کشاورزی» در UI و metadata.
- تصمیم درباره لوگوی نهایی و نیاز به نسخه SVG/transparent/cropped.
- تایید اینکه تغییرات داخلی enum/router فعلا انجام نمی شود.
- تایید اینکه ادعاهای عددی فقط از داده واقعی یا seed تایید شده خوانده شود.

### خروجی
- جدول تصمیم برند.
- لیست واژگان نهایی.
- تایید دارایی لوگو یا brief طراحی لوگوی جدید.

### Definition of Done
- Product/کارفرما نام، واژگان و لوگوی موقت/نهایی را تایید کرده باشند.

## فاز 1: پاکسازی محتوایی و ارتباطات بیرونی
هدف: حذف ردپای مستقیم برند قبلی بدون تغییر ساختار فنی پرریسک.

### کارها
- به روزرسانی `apps/web/index.html` برای title و description.
- اصلاح header و brand در صفحات auth، about، contact، news و layout احراز شده.
- اصلاح متن های صفحه اصلی از تجارت عمومی به اکوسیستم کشاورزی.
- اصلاح quick prompts و empty stateهای AI در frontend.
- بازنویسی `SYSTEM_PROMPT` در سرویس AI با guardrail کشاورزی.
- اصلاح subject و CTA ایمیل ها.
- جست وجوی کامل عبارت های قدیمی و ثبت موارد باقی مانده در backlog.

### خروجی
- UI دیگر نام «تجارت هوشمند» را در سطح قابل مشاهده نشان ندهد.
- ایمیل و AI با نام و دامنه جدید صحبت کنند.

### Definition of Done
- جست وجو برای عبارت های `تجارت هوشمند`، `مرکز تجارت متمرکز` و معادل های انگلیسی در فایل های UI/AI/email نتیجه قابل توجیه داشته باشد.
- build وب و سرور پاس شود.

## فاز 2: طراحی tokenهای بصری و سیستم پایه
هدف: تغییر تم از آبی/امبر تجاری به هویت کشاورزی مدرن و قابل نگهداری.

### کارها
- به روزرسانی CSS variables در `apps/web/src/styles/global.css`.
- تعریف tokenهای brand/agri/forest/wheat/data/error.
- اصلاح button، badge، progress، dialog و card برای هماهنگی با تم جدید.
- حذف تدریجی رنگ های hardcoded آبی/امبر/نارنجی از فایل های پرتکرار.
- تعریف کلاس های reusable برای hero، page shell، stat card و section band در صورت نیاز.

### خروجی
- theme مرکزی و قابل توسعه.
- کاهش وابستگی صفحات به کلاس های hardcoded رنگ.

### Definition of Done
- صفحه های اصلی و auth بدون رنگ آبی/امبر غالب رندر شوند.
- کامپوننت های پایه در light/dark mode خوانا باشند.

## فاز 3: بازطراحی صفحات عمومی
هدف: ایجاد تجربه عمومی متفاوت از محصول قبلی و نزدیک به زبان داده محور Tridge-inspired.

### کارها
- بازسازی صفحه اصلی با ساختار hero داده محور، چالش/راهکار، گروه های کاربری و CTA.
- بازنویسی about با ماموریت سامانه ذینفعان کشاورزی.
- بازطراحی contact با فرم رسمی، survey و پیام های شفاف.
- اصلاح news/market-insights/harvest-calendar برای یکپارچگی visual و محتوایی.
- اصلاح catalog عمومی با فیلترهای کشاورزی و نام «بازار محصولات».

### خروجی
- public surface یکپارچه، کشاورزی و مستقل از تاجر هوشمند.

### Definition of Done
- اسکرین شات desktop و mobile از صفحات عمومی با نام جدید، تم جدید و بدون ردپای مستقیم برند قبلی تایید شود.

## فاز 4: بازطراحی احراز هویت و layout کاربر
هدف: حذف حس قدیمی از اولین تجربه کاربر عضو.

### کارها
- بازطراحی login/register/login-otp با نام، لوگو و رنگ جدید.
- ساده سازی نقش ها در UI ثبت نام بدون شکستن enumهای داخلی.
- بازطراحی سایدبار و هدر موبایل با تم forest/green.
- بازنام گذاری منوها: درخواست/فرصت، بازار محصولات، مجوزهای من، تقویم برداشت، تحلیل بازار.
- بازبینی mobile bottom nav برای workflowهای اصلی.

### خروجی
- تجربه ورود و ناوبری احراز شده با برند جدید هماهنگ شود.

### Definition of Done
- کاربر پس از ورود هیچ عنوان مستقیم از برند قبلی نبیند.
- navigation روی موبایل و دسکتاپ بدون overflow و با آیکون های مناسب کار کند.

## فاز 5: بازطراحی داشبورد و جریان های عملیاتی کاربر
هدف: تبدیل داشبورد از فضای تجاری عمومی به cockpit عملیاتی کشاورزی.

### کارها
- بازطراحی dashboard با محور وضعیت پروفایل، مجوز، محصولات، درخواست ها، اعلان ها و داده های فصلی.
- اصلاح پیشنهادهای AI بر اساس پروفایل کشاورزی و نه صرفا RFQ/Marketplace.
- بازطراحی محصولات/خدمات برای تفکیک بهتر عرضه محصول و خدمت تخصصی.
- بازطراحی درخواست/فرصت برای سناریوهای خرید، فروش، مشاوره، خدمت و همکاری.
- اصلاح notifications/downloads/finance/support برای واژگان جدید.

### خروجی
- داشبورد قابل استفاده روزانه، فشرده، حرفه ای و کم تزئین.

### Definition of Done
- سناریوهای اصلی کاربر: تکمیل پروفایل، بارگذاری مجوز، ثبت محصول، ثبت درخواست و مشاهده فرصت قابل انجام و خوانا باشد.

## فاز 6: بازطراحی پنل ادمین
هدف: آماده سازی پنل برای عملیات واقعی MVP.

### کارها
- بازطراحی admin overview با گزارش های پایه و queueهای عملیاتی.
- بازطراحی taxonomy manager برای دسته بندی شغلی و محصولی.
- بازطراحی document review برای اعتبارسنجی اولیه.
- بازطراحی harvest و market content management.
- اصلاح labelهای admin users/products/ads/support/circulars برای دامنه کشاورزی.

### خروجی
- پنل ادمین از حالت عمومی تجارت خارج و به عملیات سامانه کشاورزی نزدیک شود.

### Definition of Done
- ادمین بتواند مهم ترین عملیات MVP را بدون ابهام واژگانی انجام دهد.

## فاز 7: QA، تست بصری و hardening انتشار
هدف: جلوگیری از regression فنی و بصری پس از تغییرات گسترده UI.

### کارها
- اجرای build وب و سرور.
- اجرای تست های unit/integration مرتبط با تغییرات.
- اجرای E2E smoke روی auth، dashboard، catalog، profile، documents و admin.
- گرفتن اسکرین شات desktop و mobile از صفحات کلیدی.
- جست وجوی نهایی واژگان برند قبلی.
- بررسی contrast، overflow، RTL و متن های طولانی.

### خروجی
- بسته آماده دمو/staging.

### Definition of Done
- buildها پاس شوند، blocker بصری باز نماند و snapshotهای اصلی تایید شوند.

## معیارهای پذیرش عمومی
- هیچ عنوان مستقیم «تاجر هوشمند»، «تجارت هوشمند» یا «مرکز تجارت متمرکز هوشمند ایرانیان» در UI، metadata، AI prompt یا ایمیل باقی نماند مگر به عنوان legacy/internal note مستند.
- نام نمایشی «سامانه یکپارچه ذینفعان حوزه کشاورزی» در نقاط اصلی تجربه کاربر استفاده شود.
- اصطلاحات RFQ/Marketplace/trade در UI فارسی با واژه های مصوب جایگزین شوند.
- theme غالب از آبی/امبر/نارنجی محصول قبلی فاصله بگیرد و به forest/green/data-oriented منتقل شود.
- صفحات عمومی از نظر ترکیب بندی، محتوا و حس بصری شبیه نسخه قبلی محصول نباشند.
- از Tridge فقط در سطح الگو و حس کلی الهام گرفته شود؛ متن، عدد، تصویر، کد و چیدمان دقیق آن کپی نشود.
- routeها و مدل های داخلی پرریسک بدون تصمیم جداگانه تغییر نکنند.
- build و تست های ضروری پاس شوند.
- اسکرین شات desktop و mobile برای صفحات کلیدی تایید شود.


## سوال های باز قبل از شروع پیاده سازی
- آیا استفاده از عبارت «جهاد کشاورزی» در title و متن های اصلی از نظر حقوقی/برندی تایید شده است یا باید فقط «حوزه کشاورزی» استفاده شود؟
- لوگوی ارائه شده برای نسخه نهایی تایید است یا باید فقط placeholder باشد؟
- زبان انگلیسی و عربی در مرحله اول باید کامل بازنویسی شود یا فارسی اولویت دارد؟
- آمارهای hero از داده واقعی staging خوانده شود یا seed نمایشی تایید شده تهیه کنیم؟
- آیا routeهای `/rfq` و `/marketplace` در آینده باید rename شوند یا فقط label آنها کافی است؟

## توصیه اجرایی نهایی
مسیر کم ریسک این است که ابتدا برند و واژگان قابل مشاهده پاکسازی شوند، سپس theme مرکزی تغییر کند و بعد صفحه به صفحه بازطراحی انجام شود. تغییرات عمیق مدل داده، enumها، routerها و URLها بهتر است در یک فاز جداگانه و پس از تثبیت MVP انجام شود. با این روش، نسخه جهاد کشاورزی سریع تر از محصول قبلی فاصله می گیرد، اما ریسک شکستن هسته فنی فعلی کنترل می شود.
