# نقشه راه جامع پیاده سازی پلتفرم یکپارچه وب و اندروید

## پروژه
پلتفرم یکپارچه خدماتی وب و اندروید برای پردیس نمای ویرا

## مشخصات قراردادی
| مورد | مقدار |
|---|---|
| شماره قرارداد | CRM1405001 |
| جمع ساعات مبنای پیاده سازی | 1860 ساعت برای برنامه نویس سینیور |

## هدف سند
این سند به عنوان نقشه راه اجرایی پروژه تهیه شده تا مبنای شروع پیاده سازی، مدیریت فازها، کنترل دامنه، اولویت بندی ماژول ها و تعریف معیارهای تحویل باشد. هدف این سند صرفا توصیف ایده نیست، بلکه تبدیل ایده به مسیر اجرایی قابل عمل است.

## خروجی نهایی مورد انتظار
- یک وب اپ مدیریتی و خدماتی با Next.js + tRPC + Node.js + PostgreSQL
- یک اپ اندروید Flutter متصل به همان API و دیتای مشترک
- همگام سازی واقعی داده و تنظیمات بین پنل وب و اپ اندروید
- لاگ خطا و Crash Reporting یکپارچه با GlitchTip
- قابلیت توسعه مرحله ای برای خدمات، خبر، غرفه، کیف پول، قوانین، آموزش و مشاوره
- پوشش جریان های قابل مشاهده فعلی مانند پروفایل، خبر، خدمات، غرفه، کیف پول، پشتیبانی و اطلاع رسانی

## اصول اجرایی پروژه
- اجرای مرحله ای و قابل سنجش به جای توسعه یکباره
- تمرکز بر API مشترک به عنوان هسته همگام سازی
- کاهش ریسک با اجرای Pilot رایگان و سپس توسعه اصلی
- طراحی ماژولار برای پذیرش تغییرات دامنه از سمت کارفرما
- تعریف دقیق معیار پذیرش برای هر فاز و هر ماژول
- جداسازی نیازهای قطعی از نیازهای احتمالی/آتی

## معماری کلان

### وب
- Next.js App Router
- TypeScript
- tRPC Client/Server
- پنل مدیریتی و صفحات عملیاتی

### سرور
- Node.js
- tRPC Router Layer
- Service Layer برای منطق دامنه
- Prisma ORM
- PostgreSQL
- Storage abstraction برای فایل ها و رسانه ها

### اندروید
- Flutter
- Data layer مبتنی بر REST/tRPC bridge یا HTTP client استاندارد
- State management یکدست در کل اپ
- Error reporting و release tracking متصل به GlitchTip/Sentry-compatible endpoint

### مشاهده پذیری و مانیتورینگ
- GlitchTip برای خطاهای وب و اندروید
- Structured logging برای backend
- Healthcheck برای سرویس های اصلی
- تفکیک محیط ها: dev / staging / production

| موضوع | تصمیم/پاسخ فعلی | وضعیت |
|---|---|---|

| 3. Package manager | pnpm برای monorepo | مصوب داخلی |

| 5. سرور و استقرار | فرض اجرایی این سند: سرور باید قبل از Pilot در دسترس باشد. سیستم عامل پیشنهادی Ubuntu 22.04 LTS یا Ubuntu 24.04 LTS است. استقرار ترجیحا Docker-based انجام می شود و در صورت نبود Docker، نصب و آماده سازی آن جزو چک لیست قبل از Pilot خواهد بود | نیازمند تایید کارفرما |
| 6. GlitchTip | پیشنهاد نهایی: Self-hosted روی سرور کارفرما، چون در پروپوزال نیز GlitchTip به عنوان زیرساخت متعلق به کارفرما دیده شده است | مصوب داخلی، نیازمند تایید زیرساخت |
| 7. اپ Flutter | با توجه به وجود اپ فعلی و اسکرین شات های موجود، فرض این سند شروع از «Audit + Rescue/Migrate» است نه فرض توسعه کاملا از صفر. اگر بعد از بررسی کد مشخص شود کیفیت فنی پایین یا migration پرریسک است، بازنویسی کنترل شده انجام می شود | نیازمند بررسی کد موجود |
| 8. پنل مدیریتی | در حال حاضر UI نهایی آماده نیست و در فاز 2 طراحی می شود. برای پیاده سازی پنل، انتخاب پیشنهادی: shadcn/ui به عنوان component foundation در وب | مصوب داخلی، نیازمند تایید UX |

### توضیح اجرایی برای احراز هویت
- برای کاربران نهایی، مسیر پیشنهادی احراز هویت SMS OTP است چون با ماهیت اپ موبایل و تجربه کاربری سریع سازگارتر است.
- برای نقش های مدیریتی، بهتر است ورود از مسیر جداگانه و امن تر انجام شود و به OTP عمومی کاربر نهایی وابسته نباشد.
- اگر کارفرما بخواهد، مدل ترکیبی نیز قابل اجرا است: OTP برای کاربر نهایی + Credential-based login برای ادمین.

### توضیح اجرایی برای REST + OpenAPI در Flutter
- Backend می تواند در لایه داخلی خود tRPC-محور بماند.
- برای Flutter، خروجی پایدار و قابل نگهداری تر ارائه می شود: REST endpoints + OpenAPI spec.
- این تصمیم وابستگی Flutter به bridgeهای غیرضروری را کم می کند و تست/نسخه بندی API را ساده تر می سازد.

### توضیح اجرایی برای وضعیت اپ فعلی
- تا وقتی سورس فعلی بررسی نشده، نباید روی «ادامه مستقیم از همان کد» قفل شد.
- تصمیم پایه این سند: ابتدا Audit فنی روی اپ موجود، سپس انتخاب یکی از دو مسیر:
  - Rescue/Migrate اگر ساختار فعلی قابل نجات باشد
  - Rewrite کنترل شده اگر بدهی فنی بالا باشد

### چک لیست تایید قبل از شروع Pilot
- دسترسی واقعی به سرور
- تایید سیستم عامل سرور
- تایید وجود یا عدم وجود Docker و Docker Compose
- دسترسی به سورس فعلی اپ Flutter در صورت وجود
- تایید مدل احراز هویت نهایی
- تایید Self-hosted بودن GlitchTip
- تایید انتخاب component foundation پنل وب

## فازبندی اجرایی

## فاز 0: آماده سازی و نجات پروژه
هدف: تبدیل وضعیت مبهم فعلی به نقشه اجرایی کنترل شده.

### کارها
- جمع آوری همه نیازهای قطعی و نیمه قطعی
- بررسی وضعیت فعلی اپ/اسکرین ها/جریان ها
- استخراج قابلیت های موجود فعلی از اپ (خبر، خدمات، کیف پول، غرفه، پروفایل، پشتیبانی)
- استخراج لیست ماژول های واقعی محصول
- تعیین اقلام داخل MVP و خارج MVP
- تعیین نقش های کاربری و سطح دسترسی
- تعیین سناریوهای دمو اولیه
- تعریف محیط های توسعه و استقرار

### خروجی ها
- دامنه نسخه Pilot
- دامنه نسخه MVP
- لیست موجودیت ها و نقش ها
- چک لیست فنی زیرساخت
- سناریوهای پذیرش Pilot

### Definition of Done
- دامنه Pilot و MVP به تایید داخلی برسد
- نقش ها و سناریوهای دمو نهایی شوند
- محیط توسعه قابل اجرا باشد

## فاز 1: Pilot رایگان اثبات توانمندی
هدف: اثبات فنی همگام سازی وب و اندروید روی زیرساخت کارفرما.

### قابلیت های الزامی Pilot
- وب بالا بیاید و به backend مشترک متصل باشد
- اندروید به همان backend متصل باشد
- یک یا چند موجودیت از وب قابل ویرایش باشند
- همان تغییر در اندروید بدون انتشار مجدد اپ قابل مشاهده باشد
- GlitchTip برای هر دو کلاینت راه اندازی و تست شود
- گیت تکمیل پروفایل برای دسترسی به بخشی از خدمات در Pilot قابل نمایش باشد

### سناریوهای دمو پیشنهادی
| سناریو | اقدام در وب | نتیجه در اندروید |
|---|---|---|
| خبر | ایجاد/ویرایش خبر | نمایش خبر جدید/به روز شده |
| خدمت | تغییر عنوان یا وضعیت یک خدمت | نمایش همان تغییر در لیست خدمات |
| غرفه | ایجاد یا ویرایش غرفه | نمایش غرفه در اپ |
| تنظیمات محتوا | فعال/غیرفعال کردن یک آیتم | بازتاب تغییر در اپ |
| گیت پروفایل | تغییر وضعیت تکمیل پروفایل کاربر | فعال/غیرفعال شدن دسترسی به بخشی از خدمات |

### خروجی های فنی Pilot
- محیط staging روی سرور کارفرما
- نسخه APK یا build قابل نصب برای اندروید
- پنل وب پایه برای تغییر محتوای دمو
- GlitchTip متصل و تست شده

### معیار پذیرش Pilot
- حداقل 3 سناریوی Sync با موفقیت اجرا شود
- لاگ خطا از هر دو کلاینت ثبت شود
- بدون استقرار مجدد اندروید، تغییرات محتوایی از وب دیده شود

## فاز 2: تحلیل دقیق و طراحی محصول
هدف: تبدیل نیازها به طرح اجرایی قطعی پیش از توسعه اصلی.

### خروجی ها
- Site Map کامل وب
- Feature Map کامل اندروید
- User Flow برای نقش های اصلی
- Admin Flow
- طراحی UI/UX صفحات کلیدی
- اولویت بندی بک لاگ

- دامنه غرفه و ثبت نام غرفه
- جستجو، فیلتر و موقعیت مکانی در ماژول غرفه

| تنظیمات عمومی | تنظیمات اپ/وب، نسخه، پیام های سیستمی، لینک کانال ها |

### خروجی ها
- Backend پایدار نسخه 1

- اتصال Flutter به API تولیدی
- مستندات اولیه API داخلی

## فاز 4: پیاده سازی ماژول های عملیاتی
هدف: پیاده سازی ماژول های اصلی مورد استفاده کاربر.

### ماژول خدمات
- دسته بندی خدمات
- لیست خدمات
- جزئیات خدمت
- فرم ثبت درخواست
- وضعیت درخواست
- پیگیری درخواست ها در پنل
- تعریف و دسته بندی دقیق خدماتی مانند کالاهای اساسی، بیمه، درمان، مشاوره، سفر، تفریح و سرگرمی
- پیش بینی خدمات ثبتی، تغییرات شرکت ها، کارت بازرگانی و خدمات حقوقی/داوری در مدل داده و پنل

### ماژول خبر و رسانه
- انتشار خبر
- مشاهده لیست و جزئیات خبر
- ثبت بازدید
- اشتراک گذاری
- بنرهای صفحه اول
- خبر ویژه، خبر سنجاق شده و اولویت نمایش

### ماژول غرفه
- ثبت نام غرفه
- مدیریت اطلاعات غرفه
- نمایش لیست غرفه ها
- نمایش جزئیات غرفه
- معرفی محصول/خدمت در غرفه
- جستجو در غرفه ها
- فیلتر شهری/موقعیتی در صورت فعال بودن داده مکانی
- بخش «غرفه من» برای مدیریت غرفه متعلق به کاربر

### ماژول قوانین و اسناد
- انتشار PDF/فایل/متن
- دسته بندی اسناد
- نسخه بندی و تاریخ انتشار

### ماژول آموزش و رادیو/پادکست
- لیست برنامه ها/دوره ها
- جزئیات محتوا
- پخش رسانه در اپ/وب
- مدیریت محتوا در پنل

### ماژول پشتیبانی و ارتباط با کاربر
- لینک پشتیبانی و مسیر تماس
