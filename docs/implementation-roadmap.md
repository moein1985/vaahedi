# نقشه راه پیاده سازی تغییرات مورد نیاز

## هدف
این سند ترتیب اجرای تغییرات ثبت شده در گزارش مشکلات را مشخص می کند تا تیم بتواند با کمترین ریسک، سریع ترین خروجی قابل ارائه را تحویل دهد.

## اصول اجرا
- هر فاز باید خروجی قابل تست داشته باشد.
- تغییرات حساس با Feature Flag یا حداقل Guard سمت UI/Backend اجرا شوند.
- بعد از هر فاز، تست دستی سناریوهای اصلی و تست خودکار مسیرهای بحرانی انجام شود.
- قبل از شروع هر فاز، Definition of Done همان فاز تایید شود.

## وضعیت اجرا (شروع پیاده سازی)
- انجام شد: یکدست سازی نمایش خطاهای احراز هویت در UI بر اساس کدهای UNAUTHORIZED/FORBIDDEN.
- انجام شد: نرمال سازی ورودی website (افزودن خودکار https در صورت نبود schema) در shared schema و فرم پروفایل.
- انجام شد: نرمال سازی فیلدهای اختیاری پروفایل (تبدیل رشته خالی به undefined) پیش از ارسال به API.
- انجام شد: ادامه فاز 1 برای همراستا کردن متن خطاهای ورود/عدم احراز هویت در مسیرهای باقیمانده.
- انجام شد (فاز 2): اعتبارسنجی cross-field برای تاریخ‌ها (تاریخ انقضا باید بعد از تاریخ تولید باشد).
- انجام شد (فاز 2): تفکیک نقش در صفحه ورود (دو بخش: فروشنده / خریدار).
- انجام شد (فاز 2): افزودن قابلیت تغییر زبان در منطقه احراز هویت (دسکتاپ و موبایل).
- انجام شد (فاز 3): اعمال محدودیت MIME/Size برای آپلود مدارک پروفایل و رسانه محصول (سمت UI و Backend).
- انجام شد (فاز 3): بهبود جریان آپلود با بررسی نتیجه درخواست PUT و ثبت metadata واقعی فایل (mimeType/fileSize).
- انجام شد (فاز 3): ثبت لاگ ساختاریافته برای خطاهای آپلود (invalid mime / file too large / storage failure).
- انجام شد (فاز 4): رفع UX فیلتر موبایل کاتالوگ (drawer با overlay پایدار، بستن با Escape، جلوگیری از اسکرول پس‌زمینه).
- انجام شد (فاز 4): بهبود ریسپانسیو صفحات کاتالوگ و پروفایل برای عرض‌های اندرویدی (چینش mobile-first).
- انجام شد (فاز 2): اجبار آپلود مدارک ضروری در تکمیل پروفایل (اعتبارسنجی سمت UI و Backend با پیام خطای شفاف).
- انجام شد (فاز 2): شفاف‌سازی مقصد درخواست تجاری با افزودن فیلدهای کد کالا/خدمات و مبدا تامین (شرکت/کارخانه).
- انجام شد (فاز 2): تکمیل نهایی ترتیب فیلدهای فرم ثبت محصول (سه مرحله: تعریف → تحویل/پرداخت → جزئیات).
- انجام شد (فاز 3): بهبود جریان آپلود رسانه محصول (پیش‌نمایش تصاویر، آیکن برای PDF/ویدیو، حالت Retry برای فایل‌های ناموفق).
- انجام شد (فاز 4): ایجاد فایل متمرکز form constants برای یکسان‌سازی label‌ها، placeholder‌ها، و پیام‌های خطا در سراسر فرم‌ها.
- انجام شد (فاز 2/4): استفاده form constants در صفحات profile و trade برای یکسان‌سازی نهایی متن‌های فرم.
- انجام شد (فاز 6): ایجاد تست‌های E2E جامع برای مسیرهای بحرانی (ورود/ثبت‌نام، تکمیل پروفایل، ثبت محصول، ثبت درخواست، آپلود فایل).
- انجام شد (فاز 6): اجرای Regression روی تست‌های Auth Router با DB تست محلی (پاس کامل در `auth.router.test.ts` و `__tests__/auth.test.ts`).
- انجام شد (فاز 6): پایدارسازی `e2e/critical-paths.spec.ts` با هماهنگ‌سازی selectorها و assertionها با UI فعلی؛ اجرای نهایی ۷ سناریوی بحرانی با نتیجه پاس کامل.
- انجام شد (فاز 5): اعمال Password Policy قوی در ثبت‌نام/بازیابی رمز (حداقل ۸ کاراکتر + حرف بزرگ/کوچک + عدد + نماد) و افزودن راهنمای UX در فرم ثبت‌نام.
- انجام شد (فاز 2): پیاده‌سازی مدل DB-backed برای شفاف‌سازی مقصد درخواست تجاری (فیلدهای serviceCode، supplySourceType، supplySourceName + migration + به‌روزرسانی backend/frontend).

## فاز 0: آماده سازی و هم راستاسازی (1 تا 2 روز)

### کارها
- تبدیل لیست مشکلات به Backlog قابل ردیابی با اولویت P0/P1/P2.
- تعریف معیار پذیرش برای هر آیتم (UI, UX, Validation, Auth, Upload).
- ثبت مالک هر بخش: Web, API, DB, QA.
- تعریف محیط تست یکپارچه برای بازتولید خطاها.

### خروجی
- Backlog نهایی با اولویت و مالک.
- چک لیست پذیرش برای هر آیتم.

## فاز 1: تثبیت هسته سیستم (P0) (3 تا 5 روز)

### دامنه
- خطاهای احراز هویت و پیام های اشتباه (عدم ورود/عدم احراز هویت).
- خطای اجباری بودن https در جریان های ورود و تکمیل پروفایل.
- باگ های مسدودکننده فرم ها (ثبت محصول/ثبت درخواست).

### کارها
- یکپارچه سازی Handling خطاهای احراز هویت در Frontend و Backend.
- استانداردسازی Error Code و Error Message برای سناریوهای 401/403/Session Expired.
- اصلاح Redirect/Protocol Handling تا کاربر وسط جریان با خطای https متوقف نشود.
- بازبینی گاردهای API برای اینکه علت خطا دقیق و قابل فهم باشد.

### معیار پذیرش
- کاربر لاگین نشده پیام معتبر و یکدست ببیند.
- کاربر لاگین شده ولی احراز نشده پیام متفاوت و صحیح دریافت کند.
- مسیر تکمیل پروفایل و ثبت محصول بدون بن بست پروتکل اجرا شود.

## فاز 2: پایداری فرم ها و اعتبارسنجی داده (P0/P1) (4 تا 6 روز)

### دامنه
- اعتبارسنجی تاریخ ها (تولید/انقضا).
- ترتیب و اجباری بودن فیلدهای فرم ها.
- تعیین واحد مقدار کالا و رفع ابهام مقصد درخواست.

### کارها
- پیاده سازی اعتبارسنجی Cross-field تاریخ ها سمت UI و Backend.
- اعمال ترتیب صحیح فیلدها در فرم ثبت محصول.
- افزودن اجبار آپلود مدارک ضروری در تکمیل پروفایل (با پیام خطای واضح).
- بازطراحی مدل ثبت درخواست برای شفاف سازی مقصد:
  - افزودن شناسه کالا/کد کالا.
  - افزودن مبدا تامین (شرکت/کارخانه).
- یکسان سازی متن و Label های فارسی در فرم ها.

### معیار پذیرش
- امکان ثبت داده نامعتبر از UI یا API وجود نداشته باشد.
- خطاهای فرم دقیق، فارسی و قابل اقدام باشند.
- سناریوی ثبت درخواست از ابتدا تا انتها بدون ابهام قابل انجام باشد.

## فاز 3: آپلود و رسانه ها (P1) (2 تا 4 روز)

### دامنه
- خطای آپلود PDF.
- افزودن/اصلاح ورودی های عکس، ویدیو، مدارک تخصصی.

### کارها
- بازبینی محدودیت فایل (Size/Mime/Timeout) در UI و Backend.
- رفع خطای آپلود PDF با ثبت Log ساختاریافته.
- افزودن ورودی های Upload مورد نیاز در فرم های ثبت محصول.
- افزودن پیش نمایش ساده فایل و وضعیت Upload.

### معیار پذیرش
- PDF معتبر بدون خطا آپلود شود.
- کاربر وضعیت موفق/ناموفق Upload را شفاف ببیند.

## فاز 4: UI/UX و دسترس پذیری (P1) (4 تا 7 روز)

### دامنه
- ریسپانسیو نبودن صفحات در اندروید.
- سیاه شدن صفحه هنگام فیلتر کالا.
- خوانایی پایین متن ها و جایگذاری نادرست برخی عناصر.
- ناسازگاری زبان در صفحات مختلف.

### کارها
- رفع Overlay/Stacking Context در فیلتر کالا.
- اصلاح نقاط شکست ریسپانسیو برای عرض های رایج اندروید.
- بهبود کنتراست و تایپوگرافی برای حفظ خوانایی.
- جابه جایی عناصر نادرست (مثل آیکون عضویت) طبق UX مورد توافق.
- یکسان سازی i18n و Label ها بین پروفایل و داشبورد.
- افزودن قابلیت تغییر زبان پس از ورود.

### معیار پذیرش
- مسیرهای اصلی روی موبایل اندروید بدون شکست UI اجرا شوند.
- شاخص های خوانایی متن و کنتراست در حد قابل قبول باشد.

## فاز 5: جریان ورود مبتنی بر نقش و امنیت (P1/P2) (3 تا 5 روز)

### دامنه
- تفکیک ورود فروشنده/خریدار.
- سخت گیری رمز عبور.
- فیلد پاسپورت و قوانین سایت.

### کارها
- افزودن گام انتخاب نقش در شروع ورود.
- اعمال Password Policy قوی (حروف بزرگ/کوچک/عدد/نماد).
- افزودن فیلد پاسپورت در فرم ثبت نام (در صورت نیاز کسب و کار).
- افزودن نمایش و پذیرش قوانین سایت.

### معیار پذیرش
- کاربر قبل از ورود نقش خود را مشخص کند.
- رمزهای ضعیف رد شوند و پیام راهنما نمایش داده شود.

## فاز 6: تست، سخت سازی و انتشار (2 تا 4 روز)

### کارها
- افزودن/به روزرسانی تست های E2E برای مسیرهای بحرانی:
  - ورود و احراز هویت
  - تکمیل پروفایل
  - ثبت محصول
  - ثبت درخواست
  - آپلود فایل
- اجرای Regression Test روی Web و API.
- آماده سازی Release Note و چک لیست Rollback.

### معیار پذیرش
- تست های بحرانی پاس شوند.
- مورد بحرانی باز (P0) باقی نماند.

## اولویت بندی پیشنهادی
- موج اول انتشار: فاز 1 + بخش های حیاتی فاز 2
- موج دوم انتشار: فاز 3 + فاز 4
- موج سوم انتشار: فاز 5 + فاز 6

## وابستگی ها
- تغییرات مقصد درخواست و کد کالا نیازمند هماهنگی Product و API Contract است.
- اجبار مدارک و فیلد پاسپورت نیازمند تایید حقوقی/کسب و کار است.
- تغییر نقش ورود نیازمند بازبینی قیف تبدیل و متن های UX است.

## ریسک ها و کاهش ریسک
- ریسک: افزایش خطاهای ورود بعد از تغییر Auth.
  - کاهش ریسک: Feature Toggle و انتشار تدریجی.
- ریسک: شکست UI روی دستگاه های خاص.
  - کاهش ریسک: ماتریس تست دستگاه های اندرویدی هدف.
- ریسک: ناهماهنگی متن ها در i18n.
  - کاهش ریسک: بازبینی متمرکز محتوا قبل از انتشار.

## برنامه اجرایی هفته اول
- روز 1: فاز 0 کامل + شروع فاز 1
- روز 2: تکمیل فاز 1 + تست بحرانی ورود
- روز 3: شروع فاز 2 (اعتبارسنجی تاریخ و فرم)
- روز 4: تکمیل بخش های حیاتی فاز 2 + بازبینی با QA
- روز 5: انتشار موج اول + پایش خطاها

## Definition of Done کلی
- معیار پذیرش هر آیتم تایید شده باشد.
- لاگ های خطای بحرانی پایدار شده باشند.
- تست های مسیر بحرانی پاس شوند.
- مستندات تغییرات و نکات پشتیبانی به روز شده باشند.
